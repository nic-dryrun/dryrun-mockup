1 Executive Summary

DryRunSec/dryrun-api is a Go-based backend API service handling authentication, repository scanning orchestration, and finding management. Over the analysis period, 247 pull requests were scanned, producing 38 total findings — 14 currently active, 19 resolved, and 5 dismissed. The application presents a moderate risk posture with recurring SQL injection patterns in the data access layer and incomplete tenant isolation in several model methods.

Authorization Patterns

The application uses a role-based model (admin, member, viewer) enforced through middleware. However, 4 endpoints in the admin API bypass the authorization middleware by registering routes before the middleware is applied. Additionally, tenant isolation is enforced at the service layer but not at the data access layer — the DeleteDrift and UpdateScanResult model methods accept an account UUID but do not use it in database queries.

Gap identified: Model-layer tenant isolation is missing on 3 data access methods. While the service layer currently validates ownership, these methods are vulnerable to misuse if called from a new code path that skips the service layer check.

5 Application Workflow & Data Flow

The core workflow follows this pattern: GitHub webhooks trigger scan orchestration, which dispatches to analyzer agents via internal API. Agents return findings that are stored in PostgreSQL and indexed in Pinecone for semantic search. The LLM inference layer (pkg/inference/) queries Pinecone precedents to power the Insights/CSI features.

Trust Boundaries Identified

  • External → API Gateway: All user input enters through Gin handlers. Input validation is inconsistent — some endpoints use struct tags, others perform manual checks.
  • API → Database: GORM is used for most queries, but 7 raw SQL queries exist in pkg/models/ that bypass GORM’s parameterization.
  • API → LLM (OpenAI): Prompts are constructed with user-controlled finding data concatenated directly into templates. The NitpickFilterService is particularly vulnerable to prompt injection.
  • Webhook → Internal: GitHub webhook payloads are signature-verified, but Stripe webhook verification is missing on 2 endpoints.

9 Prioritized Recommendations

P0 — CRITICAL
Remediate 3 active SQL injection findings in pkg/models/. Replace raw SQL string concatenation with GORM parameterized queries or prepared statements. Affected methods: buildSearchClause, buildCategoryFilter, buildSortClause.
P0 — CRITICAL
Add tenant isolation to data access layer. Ensure all model methods that accept an account UUID actually filter queries by it. Affected: DeleteDrift, UpdateScanResult, PurgeFindingsByRepo.
P1 — HIGH
Mitigate prompt injection in NitpickFilterService. Implement structural isolation (XML tags or delimiters) for user-controlled data in LLM prompts. Do not rely on natural language instructions alone.
P1 — HIGH
Add Stripe webhook signature verification to the 2 unprotected endpoints in pkg/handlers/stripe_webhook.go. Use stripe.ConstructEvent() for all incoming Stripe payloads.
P1 — HIGH
Reduce JWT expiry from 30 days to 24 hours and implement refresh token rotation. Update pkg/middleware/auth.go token generation.
P2 — MEDIUM
Update golang.org/x/crypto to v0.31.0+ to patch CVE-2024-45337. Run go get -u golang.org/x/crypto and validate SSH transport behavior.
P2 — MEDIUM
Standardize input validation across all API endpoints. Adopt a consistent validation library (e.g., go-playground/validator) with struct tags rather than ad-hoc manual checks.
