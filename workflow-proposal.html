Internal Proposal

GitHub Comment Workflow Redesign

Ensuring DryRun Security visibility in every PR — whether findings are detected, resolved, or dismissed.

Current Behavior

The Visibility Gap

Today, DryRun's GitHub comment is only visible when there are active, unresolved findings on a non-draft PR. In all other states — including draft PRs with findings — the comment either isn't posted or gets deleted, making it impossible to tell that DryRun scanned the PR at all.

No Findings = No Comment

When a PR scan produces zero findings, no comment is posted. From GitHub, there's no evidence DryRun ran.

Resolved = Comment Deleted

When a developer fixes the vulnerability and the next scan is clean, the comment is deleted. It looks like DryRun never ran.

Dismissed = Comment Deleted

When a developer marks the last finding as a false positive or nitpick, the finding is removed and the comment is deleted entirely.

Draft PR = No Comment

When a PR is in draft mode, DryRun scans it but does not post a comment — even when findings are detected. The developer gets no feedback until the PR is marked ready for review.

Why This Matters

Security teams, engineering managers, and auditors need to see that every PR was scanned. Deleted comments leave no audit trail, make it harder to demonstrate coverage, and reduce the perceived value of DryRun in the developer workflow.

Proposed Workflow

Always-Visible Comment Lifecycle

The core change: DryRun always posts or retains a comment on every scanned PR, regardless of whether findings exist. The comment is never deleted — it transitions between states.

PR Commit Triggers Scan
DryRun scans the PR as usual on each new commit.
No Findings Detected Configurable
Post a "clean scan" comment confirming DryRun analyzed the PR and found no vulnerabilities.
New: Comment is posted with a green status — No security vulnerabilities detected
Flag: comment_on_clean_scan — admin can disable to only post when findings exist.
Findings Detected
Post the comment with finding details, severity, code snippets, and feedback commands (same as today). On draft PRs, this is also configurable.
Flag: comment_on_draft_pr — admin can disable to suppress comments on draft PRs entirely. Configurable
Developer Fixes Issue → Next Scan Clean
Update the comment to show all findings are resolved. Do not delete.
New: Comment transitions to "All Clear" state with collapsed history of what was found and fixed.
Developer Dismisses Finding (FP / Nit)
Update the comment to reflect the dismissal. Show a badge on the finding. Do not remove or delete.
New: Finding entry is visually dimmed with a False Positive or Nitpick badge. If it was the last active finding, the comment transitions to "All Clear."
All Findings Addressed Configurable
Whether fixed or dismissed, the comment persists in a resolved state showing what was found and how it was handled.
Flag: delete_resolved_comment — admin can enable to delete the comment once all findings are resolved/dismissed (restores current behavior).

Continuous Visibility

Every PR shows DryRun ran, whether findings exist or not. Proves coverage to security teams and auditors.

Audit Trail

Resolved and dismissed findings remain visible in collapsed form, creating a clear record of what was found and how it was handled.

Developer Confidence

Developers can see their fix worked. The comment confirms the finding is resolved rather than silently vanishing.

Key Design Principles

Never Delete

The comment is never deleted. It transitions between states: clean, active, resolved, dismissed.

Always Post

Every scanned PR gets a comment, even with zero findings. Confirms coverage.

Preserve History

Resolved and dismissed findings collapse but remain accessible. Creates an audit trail for compliance.

Admin Configuration

Feature Flags

The proposed workflow ships as the out-of-the-box default. Admins can adjust comment behavior per account using the feature flags below. These live in the DryRun admin panel under account settings.

comment_on_clean_scan Default: ON

Controls whether DryRun posts a comment when a scan produces zero findings.

When ON (default)
DryRun posts a "clean scan" comment on every PR it scans, even when there are no findings. Confirms to security teams and auditors that the PR was analyzed.
When OFF
No comment is posted on PRs with zero findings. DryRun only posts a comment when it detects a vulnerability. Once a finding is posted and later resolved, the rest of the proposed workflow still applies (comment updates, history retained).
Admin Panel → Account Settings → GitHub Integration → Comment Behavior
comment_on_draft_pr Default: ON

Controls whether DryRun posts comments on draft pull requests.

When ON (default)
DryRun posts comments on draft PRs the same way it does on regular PRs — including findings, clean scan confirmations, and all state transitions. Developers get security feedback early, before the PR is ready for review.
When OFF
DryRun scans draft PRs but does not post any comment. Findings are still recorded in the DryRun dashboard. When the PR is moved out of draft, the comment is posted on the next scan with any accumulated findings.
Admin Panel → Account Settings → GitHub Integration → Comment Behavior
delete_resolved_comment Default: OFF

Controls whether DryRun deletes the comment once all findings in a PR are resolved or dismissed.

When OFF (default)
The comment is never deleted. When all findings are resolved or dismissed, the comment transitions to an "All Clear" state with a collapsed history of what was found and how it was handled. Preserves the audit trail.
When ON
When all findings in a PR are resolved (via code fix) or dismissed (via FP/nit), DryRun deletes the comment entirely. This restores the current production behavior for teams that prefer a minimal GitHub footprint.

Note: This flag only affects what happens after findings are fully addressed. It does not change how findings are posted or updated while active. Even with this flag ON, the proposed workflow applies to active findings (badges, dimming on dismiss, etc.) — deletion only occurs at the final "all clear" transition.

Admin Panel → Account Settings → GitHub Integration → Comment Behavior

Flag Interactions

How the flags combine when multiple are changed from their defaults.

Configuration Resulting Behavior
All defaults
clean_scan=ON, draft=ON, delete=OFF		 Full proposed workflow. Comment on every PR in every state. Never deleted.
clean_scan=OFF No comment on PRs with zero findings. Comment appears only when a finding is detected, then follows the full lifecycle (update on resolve, retain history).
draft=OFF No comments on draft PRs. DryRun scans silently and posts findings on the next scan after the PR leaves draft status.
delete=ON Active findings behave as proposed (badges, dismiss, etc.), but once all findings are addressed the comment is deleted rather than retained.
clean_scan=OFF + delete=ON Comments only appear when findings exist and disappear when resolved. Closest to today's behavior, but with improved handling while findings are active (badges, dimming, feedback visibility).
All three changed
clean_scan=OFF, draft=OFF, delete=ON		 Mimics current production behavior on draft and clean scans, but retains the improved active-finding UX (badges, dismiss-in-place, feedback reasons). Comment deleted on resolve.
